## Problem Statement

我同时在用 Anthropic Claude 订阅（$200/mo）+ Anthropic API + OpenAI API + 国产 API（DeepSeek / Kimi / 通义 等多家），团队 15 人月花 ~$1000，账号、key、配额、调用记录极度分散：

- **入口分散**：Cursor / Cline / Claude Code / 自写脚本各自配各自的 key，加新工具 = 重复配置
- **配额浪费**：Claude Max 月度限额经常没用满，但同时另一些时段又会撞限速
- **成本不可见**：当前没有统一的"这个月各家烧了多少钱"视图
- **多模型切换贵**：想对比 Claude vs DeepSeek vs Kimi 解同一个任务时，要改 SDK / 改环境变量

期望：**一个 OpenAI-compatible 统一端点**，所有下游工具配同一个 base URL + 同一个 key，背后由网关智能路由到具体的上游账号；附带 cost tracking 和基础观测能力。

## What Makes This Cool

1. **下游工具全部解耦**：换底层模型不用改 IDE 配置，配置文件改一行就行
2. **未来可玩花样**：路由策略（按任务类型 / 按成本 / 按健康度）、prompt caching、多账号轮询的诱惑都在这层实现
3. **第一手数据**：跑一周后，自己手上就有了"AI 用量真实分布"的 ground truth——这数据本身比中转站生意值钱（可以反过来验证 v2 是否值得做）

## Constraints

- **时间预算**：M0（本周）最多投入 8 小时，M1-M2 每周 ≤ 4 小时，否则 abort 到"裸用 LiteLLM 默认"
- **不写订阅类聚合**（Claude Code 订阅多账号轮询）until 撞墙 ≥ 3 次
- **绝不对外**：即使朋友想付费也拒绝，原因见前提 1
- **绝不从零造轮子**：必须基于 LiteLLM / NewAPI / One-API 之一改

## Premises（已确认）

**前提 1**：这是 Builder / 自用工具项目。"未来对外卖"作为可能性保留，但 v1 不考虑、v2 之前不重新评估。一旦心思又开始飘到"能不能卖"，立即回到本节重读。

**前提 2**：不从零造轮子。基于 LiteLLM / NewAPI / One-API / claude-code-router 之一改，除非碰到无法绕过的硬伤。

**前提 3**：MVP 必须本周内能自己用上。Effort 立项后第一个里程碑 ≤ 7 天达到"我每天用它访问 Claude / OpenAI"。如果第 8 天还在搞架构没人在用——回滚到 LiteLLM 默认配置，承认设计过度。

## Approaches Considered

### Approach A：纯 LiteLLM Self-Host（**推荐起步**）
- **Effort**: S (人 1 天 / CC 30 分钟)
- 部署 LiteLLM Proxy → 把所有 API key（Anthropic / OpenAI / DeepSeek / Kimi / 通义）挂上去 → 拿 OpenAI-compatible 端点喂 Cursor / Cline / 自写脚本
- ✅ 一晚上跑通；社区维护成本零；功能完备（routing / fallback / cost tracking / caching）
- ❌ 不覆盖**订阅类**（Claude Code 订阅、Copilot 订阅）；UI 是 admin 面板，不是 daily-use Web 界面

### Approach B：NewAPI / One-API Self-Host
- **Effort**: S (人 1 天 / CC 30 分钟)
- 中文圈通用，UI 完善，多用户多 key 管理
- ✅ 比 LiteLLM 更友好的中文 UI；社区活跃
- ❌ 跟 A 一样不解决订阅类；架构偏中转站，自用偏重

### Approach C：LiteLLM + claude-code-router 双层
- **Effort**: M (人 3 天 / CC 2 小时)
- LiteLLM 处理 API 类；claude-code-router 专门做 Claude Code 订阅多账号轮询；中间一层薄薄的 dispatcher
- ✅ 把"订阅类聚合"这个真正难的部分用专门工具解决
- ❌ 双层维护；Claude Code 订阅多账号轮询有 ToS 灰色风险（即使自用也可能被检测）

### Approach D：LiteLLM + Obsidian 仪表盘（创意备选，可作为 A 的 v1.5）
- **Effort**: S+ (人 1.5 天 / CC 45 分钟)
- LiteLLM 跑后端；用其 cost tracking API 反向喂数据到 `Efforts/AI-Gateway/state.md` 或专门的 cost 流水文件，每天看一次"昨天哪个模型烧了多少钱"
- ✅ 用已有 PKM 系统而非新造仪表盘；和工作流耦合好
- ❌ 不解决订阅类聚合

## Recommended Approach

**Approach A → 评估升级到 D 或 C**：

- **Week 1（M0）**：部署 LiteLLM，把 API 类全挂上，自己用上。本周必须完成。
- **Week 2-3（M1）**：评估 LiteLLM 自带成本统计够不够。够 → 不动；不够 → 上 Approach D（写 PKM 集成薄壳）。
- **Week 4+（M2）**：评估是否真的需要 Claude Code 订阅多账号轮询（取决于撞 Max 限额频率）。**等真撞墙 ≥ 3 次再考虑 C**，不预先优化。

**关键自律**：**一定不能跳过 A 直接做 C**。订阅类聚合是诱人的"复杂工作"，会让人忘记 v1 的目标是"自己用上"。

## Open Questions

- LiteLLM 的部署形态：本机 Docker 还是小 VPS？取决于 4G / 移动场景使用频率
- 国产 API（DeepSeek / Kimi / 通义）的 LiteLLM provider 支持完成度（DeepSeek 应该 OK，通义可能要自定义 adapter）
- 4-24 提到的"订阅类合规"：先按下不表，等 Approach C 评估时再回来
- Cost tracking 落到哪：LiteLLM 自带 PostgreSQL backend 还是简化的 SQLite？M0 不展开

## Success Criteria

**M0（Week 1）完成定义**：
1. LiteLLM Proxy 在本机或 VPS 上跑起来，外部可访问
2. 至少接 2 个上游：Anthropic API + 1 个其他（OpenAI / DeepSeek / Kimi 任选）
3. Cursor / Claude Code / 任意一个我每天用的工具切到 LiteLLM 端点，**强制不许偷偷直连**
4. 跑满 7 天，每天至少调用 1 次

**M0 失败信号**：第 8 天还在调试架构、自己回退去直连、或不知道怎么继续——立即 abort 到默认 LiteLLM 配置 + 写复盘。

## What I noticed about how you think

- 你 4-24 在 Inbox 已经写过这个想法、列了竞品、留了"暂存问题"，今天经过 4 轮 office-hours 兜了一圈又收敛回来——这说明你的**第一直觉是对的**，但你被"开展业务"的诱惑短暂带偏了。下次类似情况：先翻 Inbox 检查有没有相关 stub，再决定要不要重新建模。
- 你说"我相信用户对 CC Max 账号的使用一定是正态分布的"——**"我相信"是危险信号**。商业模式建立在未验证假设上时，office-hours 应该立即推回去要数据。今天你诚实地说"暂没有数据"，这是好的，这种诚实把决策从"赌 $100k"拉回到"先做验证工具"。
- 你说"承担 $100k 以内损失"——你有 risk capacity，这是真实资产。但 risk capacity ≠ risk plan。下次再有商业想法，先问"我能用这 $100k 买到什么洞察"而非"我能用这 $100k 撑多久"。
- 你最终选 (1) 直接立项，没纠结"是否再观察一周"。**Bias-to-action 是好品质**，前提是 v1 scope 收得住。今天 scope 收到了"纯自用、本周可用"，这条路 bias-to-action 很合适。
