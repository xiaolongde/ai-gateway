# AI 订阅统一网关（反向代理聚合）— 原始构想

> **来源**：`Atlas/Inbox/2026-04-24-AI-统一网关.md`（Inbox 沉淀，2026-04-24 写）。
> **状态**：已被 [[2026-05-03-design]] 取代。本文档作为历史快照保留，不再迭代。
> **价值**：保留早期问题列表（已经做了竞品调研、形态选项、合规风险预判），证明今天的 office-hours 不是凭空决策——是把 4-24 留下的"暂存问题"实际答出来了。

---

把所有订阅中的 AI 服务反向代理出来，通过统一入口使用——Claude Code、OpenAI、GitHub Copilot、Gemini 等。

## 初始动机（待核实）

- 订阅分散 → 入口分散 → 切换成本
- 不同模型擅长不同任务，希望一键切换对比
- 便携：一套凭据，多终端使用

## 待想清楚

- **范围**：覆盖哪些服务？
  - API 类（Anthropic / OpenAI / Gemini API）相对容易
  - 订阅类（Claude Code 订阅 / ChatGPT Plus / Copilot 订阅）涉及 cookie/token 抓取或 MITM，合规与稳定性是重点
- **形态**：
  - (a) OpenAI 兼容 API 网关（各家转成 OpenAI Schema，客户端统一）
  - (b) Web UI 聚合器（类似 LibreChat / OpenWebUI）
  - (c) 专用客户端 + 本地 broker
- **核心价值**：
  - 统一 API 接口，下游工具（Cursor / Cline / 自写脚本）只对一个端点？
  - 省钱（路由到最便宜/够用的模型）？
  - 隐私/审计（请求流水本地留存）？
- **合规与风险**：
  - Copilot / Claude Code 订阅的 ToS 是否允许第三方代理
  - 账号风控：频繁非官方 client 访问可能封号
- **已有方案比较**（必做调研）：
  - OpenRouter（API 聚合，但订阅类覆盖不全）
  - LibreChat / OpenWebUI（UI 聚合）
  - One-API / NewAPI（中文圈常用 API 路由）
  - LiteLLM（Python 侧 API 统一）
  - 重造轮子 vs 基于现有 + 补订阅类

## 暂存问题

- 是纯**自用工具**，还是可能面向小众群体（开源/SaaS）？
- 时间投入上限是多少？
- 最小有用版本（MVP）是什么——先代理 3 家 API？还是只解一个痛点（如 Copilot 转 OpenAI 格式）？

---

## 2026-05-03 office-hours 给出的答案（指针，详见 [[2026-05-03-design]]）

| 暂存问题 | 答案 |
|---------|------|
| 自用 vs 小众群体 | **纯自用，v1 不对外** |
| 时间投入上限 | **MVP 7 天内能自己用上**，超时回退到 LiteLLM 默认配置 |
| MVP 范围 | **API 类先全（LiteLLM Approach A）**；订阅类聚合等撞墙 3 次以上再考虑 |
| 形态 | **(a) OpenAI 兼容 API 网关**，舍弃 Web UI 聚合器和本地 broker 路径 |
| 已有方案比较 | **基于 LiteLLM 改**，不从零造 |
