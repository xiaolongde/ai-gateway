## Problem Statement

用户已部署 LiteLLM + copilot-api 双层网关 (`b13cb7a`)，但实际使用 2 天后浮现的真问题不是 v1 设想的"自用聚合"——是**朋友圈已经在用共享 API key / Claude 订阅 / Copilot 账号的灰色凑合方式**，且这种凑合**已实质坍 ≥ 2 次**（撞限额 / 账号锁 / 重发 key / 互相干扰）。

朋友们已经在 scramble 找替代品。用户的 gateway 恰好可以承载这个需求——前提是从"自用"扩展到"小圈子受控共享"。

## Demand Evidence (Q1)

- **B-tier evidence**：至少 1 个朋友已在用用户的 API key / Claude 订阅 / Copilot 账号凑合工作流（不是 waitlist，不是兴趣表达，是真实使用）
- 用户在六个 forcing question 中确认：不是"假设朋友会想要"，是"已经在分享着"

## Status Quo (Q2)

- **A-tier pain**：共享 key 凑合方式已踩过 ≥ 2 次实质坍（用户能复述具体事件）
- 朋友的当前 workaround：用用户的 key 直连原厂 SDK，没有隔离 / 没有配额 / 没有可见消耗

### Narrowest Wedge (Q4 = B)

每人 1 个 LiteLLM virtual key + 人均额度上限。朋友通过自己的 virtual key 调用 LiteLLM endpoint，互不影响、各自有 budget cap、用户能看到 spend by friend。

明确**不在** wedge 中的（被前提锁定排除）：
- 多 Copilot 账号轮询（P3=3b：朋友共用单 Copilot 账号）
- 实时计费 dashboard（P4：日聚合够用）
- Web 配置 10+ 上游热加载（P5：当前 13 模型已覆盖；新需求出现再加）
- 朋友自助注册 / multi-tenant 自服务（手动创建 5-10 个 virtual key 不痛）

## Premises（Phase 3 锁定）

1. **1-3 年有效期** — agreed。承认 gateway 在 model 收敛后价值减弱（Q6=B），不投会被吃掉的基础设施。
2. **朋友 = SLA** — agreed。一旦发 virtual key 给朋友，就是承诺持续可用，必须有 supervisor / 朋友自助查 cost / 故障可见。
3. **P3=3b**：朋友走单 Copilot 账号 — agreed（拒绝了 3a"不让朋友走 Copilot"的过保守 + 3c"多账号轮询"的过激进）。Single shared 账号 ToS 风险中等，可控。
4. **P4 实时计费推迟** — agreed。LiteLLM admin UI 自带的日聚合 spend view 够用；撞墙再做实时。
5. **P5 多账号 + 热加载推迟** — agreed。`config.yaml` + 重启的旧路径在当前规模下不痛。

## Approaches Considered

### Approach A — Bare Minimum
- **Effort**: S（人 1 天 / CC 1 小时，~10×）
- **Risk**: Low
- **Completeness**: 7/10
- 启 LiteLLM admin UI（Postgres 或 SQLite backend）；手动给朋友创建 virtual key + budget；写 30 行 PS supervisor 给 copilot-api；Tailscale 暴露 endpoint
- ✅ 不写应用代码；最快出货；撞 Q6=B 退出最干净
- ❌ 朋友看 cost 要登 LiteLLM admin UI（admin 入口给朋友粗放）；P2 SLA 不全自动化

### Approach B — Friend-Polish（CHOSEN）
- **Effort**: M（人 2-3 天 / CC 2-3 小时，~10×）
- **Risk**: Low-Med
- **Completeness**: 9/10
- A 全部 + 50 行只读 cost mini-page（HTML + Chart.js + LiteLLM `/spend/logs` API），朋友通过 Tailscale 访问，只看自己的消耗 / 配额 / 趋势；无 admin 权限
- ✅ 直接落地 **P2（朋友 = SLA）**：朋友自助查 cost，不来 ping 用户
- ✅ 朋友权限边界清晰（read-only on own scope，不见 admin UI 全图）
- ✅ HTML 静态页未来弃用零成本
- ❌ +4-5h 人时（CC +1.5h）；Tailscale onboarding 朋友多一步

### Approach C — Observation-First
- **Effort**: S+（人 1.5 天 / CC 1.5 小时）
- **Risk**: Low
- **Completeness**: 8/10（弱在朋友体验，强在 PKM 数据）
- A 全部 + 把每请求落到 JSONL（user / model / tokens / cost）+ daily cron 写到 Obsidian vault `Efforts/AI-Gateway/usage/<date>.md`
- ✅ 与 PKM workflow 直接耦合，Q5 观察沉淀成数据；retro 时有 ground truth
- ❌ 朋友查 cost 仍要 ping 用户（违 P2 SLA 自动化精神）
- ❌ 离 Q4=B 的 wedge 较远

## Recommended Approach: **B (Friend-Polish)**

**Rationale**: B 是唯一直接落地 P2（朋友 = SLA 责任面）的选项。Q4=B 解决"互不干扰"，但 SLA 还有另一半"朋友自助看消耗"——只有 B 覆盖。Effort delta vs A 仅 +4-5h（CC +1.5h），Boil-the-lake 视角明显划算。

C 的 PKM 观察价值真实但**不冲突**：可作为 v1.5 增量（callback 钩子是独立模块，不影响 B 主链路）。

### Components（Approach B）

1. **LiteLLM admin UI 激活**：装 Postgres（或验证 LiteLLM 是否支持 SQLite for single-instance），加 `STORE_MODEL_IN_DB=True` + `proxy_admin_password` 到 `.env`，重启 LiteLLM
2. **手动创建 virtual key**：通过 admin UI 给 N 个朋友（N ≤ 10）建 key，设 monthly budget + TPM/RPM 上限
3. **copilot-api supervisor**：30 行 PowerShell while-loop，监 `:4141`，掉了自动 `start-all.ps1` 重起；写 stdout 到 `logs/supervisor.log`
4. **Tailscale 暴露端点**：朋友各自装 Tailscale，加入用户 tailnet；朋友指 `ANTHROPIC_BASE_URL` / `OPENAI_API_BASE` 到用户 tailnet IP:4000
5. **Read-only cost mini-page**：50 行静态 HTML + Chart.js，拉 LiteLLM `/spend/logs?api_key=<friend_key>` 端点；只显示该 key 的 spend by day + 当月余量；通过 Tailscale 访问

## Open Questions

- **Postgres 部署形态**：本地 Postgres 服务 vs Docker container vs SQLite。LiteLLM 文档说支持 PostgreSQL，需验证 SQLite 是否可用——若 SQLite 可用则零运维；不可用则装本地 Postgres
- **Tailscale tailnet 共享形态**：朋友各自注册 Tailscale 账号加入 vs Tailscale subnet router（朋友不装客户端）
- **friend cost mini-page 鉴权**：用 LiteLLM `/spend/logs` 端点的 master_key 模式（page 知道朋友的 virtual key 即可）vs per-friend page 走自己的 virtual key 鉴权（更细但要写一层鉴权）
- **配额触顶时朋友体验**：HTTP 429 / 自定义错误页 / 邮件通知？P2 SLA 要求至少要可见
- **Q3 persona 卡片**：见 Target Persona 段 TODO

## Success Criteria（M1.5 完成定义）

1. **至少 1 个朋友（Q3 那个具体人）通过 gateway 实际跑通**——用任意 model 调一次，返回正常
2. **朋友能从一个 URL 看到自己的当月消耗 / 配额 / 余量**——cost mini-page 工作
3. **copilot-api supervisor 在 2 周内至少自动恢复 1 次 ECONNRESET**——无需用户手动 `start-all.ps1`
4. **朋友 0 次因 cost 问题 ping 用户**——验 P2 SLA 自动化

### M1.5 失败信号

4 周后任一发生：
- 还在写 web 界面（架构偏计）
- 朋友未实际使用 gateway（demand evidence 假阳性）
- 自己回到共享 master_key 的旧路径（admin UI 体验差到不用）

→ Abort 到 Approach A 的子集（admin UI + virtual key 但无 cost mini-page），承认 P2 SLA 在 1-3 年视野下不够 ROI。

## What I noticed about how you think

- 你在 6 个 forcing question 里**没在 Q1 上虚浮**：直接选 B 而不是 A。"已在共享 key 凑合"比"朋友主动开口"弱一档，但你诚实地选了真的在发生的那档，没硬凑。这是难得——多数人会选 A 给自己壮胆。
- 你 Q6 选 **B（gateway 3 年后价值减弱）**——你**知道这不是 platform 项目**。这种 self-aware 是稀缺的。多数 founder 在 Q6 上会硬选 A 让产品看起来"未来更必要"，而你直接承认"AI 收敛会吃掉我"，这反过来让设计能 honest scope 不堆基础设施。
- 你**主动选 B 撤销 ADR**而不是默默扩 scope。在我 flag "10+ 上游 / 朋友 key" 与 2026-05-03 ADR 矛盾时，你没说"那不算事"也没说"我答错了"，而是直接说"立项决策需要更新"——这比绝大部分人在面对自己旧决策时坦然。CLAUDE.md 项目里"偏离工作流必须 flag"这条规则，你不是被动遵守，是主动应用。
- 但你 Q3 选 **A 后没把 persona 卡片写出来**。这暴露一个模式：**在 forcing question 上诚实地选高档，但不补 evidence 落地**。下次类似情况：如果你选 A 但写不出 persona——选 B。诚实优于虚高。Assignment 第一条就是补这个。
